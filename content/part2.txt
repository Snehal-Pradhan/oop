# Part 2 — Core Principles of OOPS

## 1. Encapsulation

Bundling data + methods together, hiding internal state via `private` fields, exposing controlled access via public getters/setters.

```java
class BankAccount {
    private String accountHolderName;
    private double balance;

    public BankAccount(String name, double balance) {
        this.accountHolderName = name;
        this.balance = balance;
    }

    public double getBalance() { return balance; }
    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    public void withdraw(double amount) {
        if (amount <= balance) balance -= amount;
    }
}
```

**Benefits:** data security, flexibility (change internals without breaking external code), modularity, easier debugging, reduced complexity.

**Note:** Encapsulation is a design principle; access modifiers + getters/setters are mechanisms to achieve it.

## 2. Access Modifiers

| Modifier | Class | Package | Subclass | World |
|----------|-------|---------|----------|-------|
| `public` | ✅ | ✅ | ✅ | ✅ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| default (none) | ✅ | ✅ | ❌ | ❌ |
| `private` | ✅ | ❌ | ❌ | ❌ |

**`public`** — accessible everywhere (APIs, global utilities)

**`private`** — only within the class (sensitive data). Encourages getter/setter pattern.

**`protected`** — same package + subclasses (even in different packages)

**default** (no modifier) — package-private only

## 3. Inheritance

A class (subclass/child) inherits attributes and methods from another class (superclass/parent).

```java
class School {
    void printSchoolName() { System.out.println("DPS"); }
}

class Student extends School {
    void printStudentName() { System.out.println("Raj"); }
}
// Usage
Student s = new Student();
s.printSchoolName();   // inherited
s.printStudentName();  // own
```

### Types of Inheritance in Java

**Single:** One parent → one child (`Animal → Dog`)
**Multilevel:** Chain of inheritance (`Animal → Mammal → Dog`)
**Hierarchical:** One parent → many children (`Animal → Dog`, `Animal → Cat`)

```java
// Single
class Dog extends Animal { }

// Multilevel
class Mammal extends Animal { }
class Dog extends Mammal { }

// Hierarchical
class Dog extends Animal { }
class Cat extends Animal { }
```

**Multiple inheritance** (a class with 2+ parents) is NOT allowed for classes in Java (diamond problem). Achieved via interfaces.

### super Keyword
```java
class Parent {
    Parent() { System.out.println("Parent"); }
}
class Child extends Parent {
    Child() {
        super();  // calls Parent constructor (implicit even if not written)
        System.out.println("Child");
    }
}
```

- `super()` calls parent constructor (must be first statement)
- `super.methodName()` calls parent's overridden method

### Method Overriding
- Same name, parameters, return type as parent
- Cannot be more restrictive access
- Use `@Override` annotation
- Only inheritable methods (public/protected) can be overridden

### Advantages of Inheritance
**Reusability** — reuse parent code, **Modularity** — separate concerns, **Extensibility** — add features without modifying parent, **Maintainability** — fix in one place

## 4. Polymorphism

"Many forms" — same entity behaves differently in different contexts.

### Compile-Time (Static) — Method Overloading
- Same class, same method name, **different parameters** (number, type, or order)
- Resolved at compile time
- Return type alone cannot differentiate

```java
class Calculator {
    int add(int a, int b) { return a + b; }
    double add(double a, double b) { return a + b; }
}
```

### Run-Time (Dynamic) — Method Overriding
- Parent reference, child object — JVM decides at runtime
- Dynamic Method Dispatch

```java
class Animal { void sound() { System.out.println("..."); } }
class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark"); }
}

Animal myAnimal = new Dog();
myAnimal.sound();  // "Bark" (runtime decision)
```

### Overloading vs Overriding

| Aspect | Overloading | Overriding |
|--------|-------------|------------|
| Dependency | Same class | Inheritance required |
| Parameters | Must differ | Must be same |
| Access modifier | Any | Cannot be more restrictive |
| Resolution | Compile-time | Run-time |
| Return type | Can differ | Must be same |

## 5. Abstraction

Hiding implementation details, showing only essential features.

### Abstract Classes
- Cannot be instantiated directly (`new`)
- Can have both abstract (no body) and concrete methods
- Subclass **must** implement all abstract methods (unless subclass is also abstract)

```java
abstract class Animal {
    void eat() { System.out.println("Eats food"); }     // concrete
    abstract void sound();                               // abstract, no body
}

class Dog extends Animal {
    @Override
    void sound() { System.out.println("Bark"); }
}
```

- Can have constructors (called via `super()` from subclass)
- Can extend another abstract class

### Interfaces
- Blueprint/contract — all methods are implicitly `public abstract` (before Java 8)
- Fields are implicitly `public static final` (constants)
- A class can `implement` **multiple** interfaces

```java
interface Animal {
    void sound();    // implicitly public abstract
    void eat();
}

class Dog implements Animal {
    public void sound() { System.out.println("Bark"); }
    public void eat() { System.out.println("Dog eats"); }
}
```

### Default Methods (Java 8+)
Allow adding methods to interfaces without breaking existing implementations.

```java
interface Example {
    default void defaultMethod() {
        System.out.println("Default implementation");
    }
}
```

### Static Methods in Interfaces
Belong to the interface, called via `InterfaceName.staticMethod()`.

### Interface Inheritance
Interfaces can extend other interfaces (`Mammal extends Animal`).

### Abstract Class vs Interface

| Aspect | Abstract Class | Interface |
|--------|---------------|-----------|
| Instantiation | ❌ | ❌ |
| Constructors | ✅ | ❌ |
| Instance variables | ✅ | ❌ (`public static final` only) |
| Methods | abstract + concrete | abstract + default + static (Java 8+) |
| Multiple inheritance | ❌ (single extends) | ✅ (multiple implements) |
| Access modifiers | Any | `public` only |

## 6. Static Keyword

Belongs to the **class**, not any instance. Loaded once in memory.

### Static Variables
Shared across all instances.

```java
class Counter {
    static int count = 0;
    Counter() { count++; }
}
// Counter.count → 2 after creating 2 objects
```

### Static Methods
Called via class name, cannot access non-static members directly.

```java
class MathUtils {
    static int add(int a, int b) { return a + b; }
}
int result = MathUtils.add(5, 3);
```

### Static Blocks
Run once when class is loaded, before any object creation.

```java
class Example {
    static int value;
    static { value = 10; }
}
```

### Interaction Rules
- Static methods can only directly access static members
- To access non-static from static: create an instance first
- Non-static methods can access static members freely

## 7. Inner Classes

Classes defined inside another class.

### Static Nested Class
- `static` modifier, no reference to outer instance
- Can only access outer's static members

```java
class Outer {
    static int staticVar = 100;
    static class StaticNested {
        void display() { System.out.println(staticVar); }
    }
}
Outer.StaticNested nested = new Outer.StaticNested();
```

### Non-Static Inner Class
- Associated with an instance of outer class
- Can access all members (including private)

```java
class Outer {
    int instanceVar = 42;
    class Inner {
        void display() { System.out.println(instanceVar); }
    }
}
Outer o = new Outer();
Outer.Inner i = o.new Inner();
```

### Local Inner Class
- Defined inside a method/block
- Accessible only within that scope
- Can only access `final` or effectively final local variables

### Anonymous Inner Class
- No name, instantiated at declaration
- Used for one-time implementations

```java
abstract class Greeting {
    abstract void sayHello();
}
Greeting g = new Greeting() {
    void sayHello() { System.out.println("Hello"); }
};
```

## 8. Class Relationships

### Association (general interaction)
Classes connected through references. Multiplicity: 1-to-1, 1-to-many, many-to-many.

### Aggregation (weak has-a)
"Whole" contains "part" but both can exist independently. Weaker bond.

```java
class Department {
    private List<Employee> employees;
    Department(List<Employee> employees) { this.employees = employees; }
}
// Employee exists even if Department is deleted
```

### Composition (strong has-a)
"Whole" owns "part" — if whole is destroyed, parts are destroyed. Stronger bond.

```java
class House {
    private List<Room> rooms;
    House() {
        rooms = new ArrayList<>();
        rooms.add(new Room("Living Room"));
    }
}
// Room is destroyed when House is destroyed
```

| Aspect | Association | Aggregation | Composition |
|--------|------------|-------------|-------------|
| Relationship | General | Weak | Strong |
| Ownership | None | One contains but doesn't own | One owns the other |
| Independence | Independent | Part can exist without whole | Part cannot exist without whole |

## 9. Object Cloning

Creating an exact copy of an object in a different memory location.

### Cloneable Interface
Marker interface (no methods). Signals JVM that `clone()` is safe to invoke.

### Shallow Cloning
Default `clone()` — copies primitives, shares references for nested objects.

```java
class Person implements Cloneable {
    String name;
    Address address;

    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();   // shallow copy
    }
}
// Changes to clonedPerson.address affect person.address (shared reference!)
```

### Deep Cloning
Creates independent copies of all nested objects.

```java
class Person implements Cloneable {
    String name;
    Address address;

    @Override
    protected Object clone() throws CloneNotSupportedException {
        Person cloned = (Person) super.clone();
        cloned.address = (Address) address.clone();  // manually clone nested
        return cloned;
    }
}
// Changes to clonedPerson.address DON'T affect person.address
```

| Aspect | Shallow | Deep |
|--------|---------|------|
| Primitive fields | Copied | Copied |
| Object references | Shared | Independent copies |
| Nested objects need clone() override? | No | Yes |
| Use case | Primitive-only fields | Mutable reference fields |

## 10. Exception Handling

Mechanism to handle runtime errors and maintain program flow.

### Try-Catch
```java
try {
    int result = 10 / 0;                     // risky code
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero");
}
// Program continues after catch
```

**Multiple catch blocks:** Order matters — catch specific exceptions before general ones.

### Finally Block
Always executes (whether exception occurs or not). Used for cleanup.

```java
try { ... }
catch (Exception e) { ... }
finally { System.out.println("Always runs"); }
```

### throw vs throws

| `throw` | `throws` |
|---------|----------|
| Inside method, explicitly throws exception | In method signature, declares possible exception |
| `throw new Exception("msg")` | `void method() throws IOException` |
| One at a time | Multiple comma-separated |

```java
void checkAge(int age) throws Exception {
    if (age < 18) throw new Exception("Not eligible");
}
```

### Custom Exceptions
```java
class CustomException extends Exception {
    CustomException(String msg) { super(msg); }
}
throw new CustomException("App-specific error");
```

### Checked vs Unchecked

| Checked | Unchecked |
|---------|-----------|
| Compiler enforces handling (try/catch or throws) | No compile-time enforcement |
| `IOException`, `SQLException` | `ArithmeticException`, `NullPointerException` |

## 11. Generics

Type safety + code reusability. Introduced in Java 5.

```java
ArrayList<String> list = new ArrayList<>();  // type-safe
list.add("Hello");
// list.add(100);  // compile-time error
String s = list.get(0);  // no casting needed
```

### Generic Classes
```java
class Box<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}
Box<Integer> intBox = new Box<>();
Box<String> strBox = new Box<>();
```

### Generic Methods
```java
public <T> void print(T data) {
    System.out.println(data);
}
```

### Bounded Type Parameters
```java
class NumericBox<T extends Number> {   // only Number subclasses allowed
    private T num;
    public double square() { return num.doubleValue() * num.doubleValue(); }
}
```

### Wildcards
- **Upper bounded** `? extends T` — read allowed, write restricted
- **Lower bounded** `? super T` — write allowed, read restricted

```java
void printList(List<? extends Number> list) { }  // accepts List<Integer>, List<Double>, etc.
void addNumbers(List<? super Integer> list) { }  // accepts List<Integer>, List<Number>, List<Object>
```

### Raw Types & Type Erasure
- Raw type: `Box box = new Box();` (no type parameter — avoid this)
- Type erasure: generic type info removed at compile-time, replaced with `Object` (or bound type)
- JVM has no knowledge of generics at runtime

## 12. File Handling

### File Class
```java
import java.io.File;
File file = new File("example.txt");
file.exists();        // boolean
file.createNewFile(); // boolean
file.delete();        // boolean
file.length();        // bytes
file.getAbsolutePath();
```

### Writing (BufferedWriter)
```java
BufferedWriter writer = new BufferedWriter(new FileWriter("output.txt"));
writer.write("Hello");
writer.newLine();
writer.close();
```

### Reading (BufferedReader)
```java
BufferedReader reader = new BufferedReader(new FileReader("example.txt"));
String line;
while ((line = reader.readLine()) != null) {
    System.out.println(line);
}
reader.close();
```

### Try-with-Resources (Auto-close)
```java
try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
} catch (IOException e) {
    e.printStackTrace();
}
// No need to manually close — resource closed automatically
```

### Logger Example
```java
class Logger {
    private String path;
    Logger(String path) throws IOException {
        File file = new File(path);
        if (!file.exists()) file.createNewFile();
        this.path = path;
    }
    public void log(String message) {
        try (BufferedWriter bw = new BufferedWriter(new FileWriter(path, true))) {
            bw.write(message);
            bw.newLine();
        } catch (Exception e) {
            System.out.println("Failed to log: " + message);
        }
    }
}
```

### Common Issues
- `FileNotFoundException` — file doesn't exist
- `IOException` — permission issues, disk full
- Resource leaks — always close streams (or use try-with-resources)
